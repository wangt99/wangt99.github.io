# 从零开始的分布式（01）

## 时间，时钟和事件顺序

本部分将研究分布式系统中的时间概念。对时间的假设构成了分布式系统模型的一个关键部分。例如，基于超时的故障检测器需要测量时间用于确定何时超时。操作系统依赖计时器和时钟，以便安排任务，跟踪CPU的使用，以及别的一些任务。应用程序经常希望记录时间发生的时间和日期：例如，当调试分布式系统中的错误时，时间戳对调试很有帮助，因为它们允许我们在同一时间不同节点上发生事件的场景。所有这些都要对时间进行精确测量。

![image-20230509163259834](/home/gurus/Code/2023/blog/assets/img/image-20230509163259834.png)

分布式系统中的时钟和时间：

+ 调度器、超时、故障检测器、重试定时器
+ 性能测量、统计、分析
+ 日志文件和数据库：记录事件发生的时间
+ 有时间限制的数据（如缓存条目）
+ 确定几个节点上的时间顺序

我们区分了两种类型的时钟：

+ 物理时钟：计算经过的秒数
+ 逻辑时钟：计算事件，例如发送的消息

### 物理时钟

物理时钟以秒为单位测量时间。它们包括基于钟摆或类似机制的模拟/机械钟，以及基于振动石英晶体的数字钟。

石英钟很便宜，但不完全准确。一个时钟运行快或慢的速度被称为漂移 drift。

漂移通过**百万分率**来衡量。$1 ppm = 1 microsecond / second = 86 ms / day = 32 s / year$。大多数计算机的时钟漂移在50ppm以内。

当需要更高的精度时，则要用到原子钟。另外一种获取高精度时间的方式依靠GPS卫星定位系统。基于原子钟（TAI）的时间测量系统运作一向良好，但与我们的日常时间脱节。

解决方案是使用世界协调时（UTC），以原子钟为基础，但考虑了地球自动变化的修正。时区被指定为UTC的偏移量。

UTC包括**闰秒**。由于闰秒的存在，一个小时不总是3600秒，一天也不总是86400秒。这使得需要处理日期和时间的软件变得复杂。

![image-20230510202447215](/home/gurus/Code/2023/blog/assets/img/image-20230510202447215.png)

为了正确计算，使用时间戳的软件需要知道闰秒的情况。比如，如果你想计算两个时间戳之间经过多长时间，需要知道这两个日期之间插入了多少个闰秒。

软件中常见的方法是直接忽略闰秒。对于只需要粗略时间的软件来说，这种方法很好。然而，操作系统和分布式系统依靠高精度的时间戳来精确测量时间，在这种情况下，一秒钟的差异都非常明显，忽略闰秒非常危险。

如今，部分软件会显式地处理闰秒，而其他程序依然会忽略它们。当前一个可行的解决方案是，当正数闰秒发生是，不是将其直接插入23:59:59和00:00:00之间，而是通过在这段时间内故意放慢时钟速度（或在负闰秒下加快时钟速度），将额外的秒数分散到该时间前后的几个小时。这种方法被称为 **semearing the leap second 涂抹闰秒**。这只是一种替代方案，但并总是可行方案。



### 时钟同步

![image-20230510203211114](/home/gurus/Code/2023/blog/assets/img/image-20230510203211114.png)

计算机采用石英钟来记录物理时间/UTC。由于石英钟漂移，误差会逐渐累积，造成时钟偏移。最常见的解决方案是使用网络时间协议（NTP），定期从更精确的时间源服务器获取当前时间。为了减少随机变化的影响，NTP对时间测量进行多次采样，并应用统计来消除异常值。

### Causality 和 happens-before

现在我们将继续讨论分布式系统中的事件排序问题，这与时间的概念密切相关。在这个情景中，用户A发送一个声明m1，并将其作为消息发送给另外两个用户B和C。收到m1时，用户B向用户A和C发送回复m2。然而，即使我们假设网络链接是可靠的，**reordering 重排**也可能发送，所以如果m1在网络中稍有延迟，C可能会在m1之前受到m2。

从C的视角来看，消息顺序是混乱的。C首先看到的是回复，然后是它所回复的信息。这看起来就像B能够看到未来，并在A发言之前就预见了A的声明。在现实生活中，这中对消息的重新排序是不会发生的，所以我们从直觉上也不希望它发生计算机系统中。

一个更技术性的例子。考虑m1是一条在数据库中创建一个对象的指令，而m2是一条更新对象的指令。如果一个节点在m1之前处理m2，它将首先尝试更新一个不存在的对象 ，然后创建一个随后不会被更新的对象。只有当m1在m2之前处理，数据库指令才有意义。

![image-20230511221252335](/home/gurus/Code/2023/blog/assets/img/image-20230511221252335.png)

C如何才能确定信息的正确顺序？单调时钟是行不通的，因为它的时间戳在不同的节点是没有可比性的。第一个可能的方案是，每当用户想发送消息时，从time-of-day时钟中获取一个时间戳，并将该时间戳附在消息上。在这种情况下，我们可以预期m2的时间戳晚于m1的回应，所以m2一定发生在m1之后。

不幸的是，在一个部分同步模型中，这并不可靠。由于NTP之类的时钟同步总是对两个时钟之间真实偏移的近似，尤其是如果网络延迟是不对称的。

![image-20230512110756450](/home/gurus/Code/2023/blog/assets/img/image-20230512110756450.png)

为了公式化 "正确" 顺序的含义，我们定义 **happen-before relation** 。假设每个节点只有一个执行线程，所以对与一个节点的任何两个执行步骤，先后执行是清楚的。同一节点上发生的事情有一个**严格全序**。一个多线程的过程可以通过使用每个线程一个节点来进行建模。

我们通过定义跨节点的顺序：一个消息应该被先发送，后被接收。我们假设每条发送的信息都是唯一的，所以当收到一条消息时，我们总是能明确知道该消息是在哪里和何时发送的。在实践中，重复消息可能存在，但我们可以使它们唯一，例如在每个消息中包括发送者节点ID和一个序列号。

最后，我们使用传递律，得到 happen-before 关系。这是一个偏序，也就是说，对于某些事件 a 和 b 来说，有可能既没有 a 在 b 之前发生，也没有 b 在 a 之前发生， 这种情况下，我们把 a 和 b 称为并发。这里的并发并不是指 "at the same time"，而是指a 和 b 是独立的，即不存在从一个到另一个的信息序列。

![image-20230512115948973](/home/gurus/Code/2023/blog/assets/img/image-20230512115948973.png)

happens-before关系是对分布式系统中因果关系的一种推理方式。

因果关系考虑的是信息是否可以从一个事件传递到另一个实践，因此一个事件是否可能影响到另一个事件。

因果关系是从物理学中借来的，人们普遍认为，信息的传播速度不可能超过光速。因此，如果有两个事件a和b，如果它们在空间上相距很远，但在时间上相距很近，那么从a 发出的信号不可能在事件b之前到达b的位置，反之亦然。因此，a和b必须是没有因果关系的。
