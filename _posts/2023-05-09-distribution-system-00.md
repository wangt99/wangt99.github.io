 # 从零开始的分布式

本系列是分布式系统学习记录，资料会参考课程 (如6.824)，博客，以及一些代码等。将以**基础理论**，**代码实践**等来构建知识体系

## 开篇

在一台计算机上的并发也被称为 **shared-memory-concurrency** 共享内存并发，因为在同一进程中运行的多个线程可以共享该进程的地址空间。因此，数据可以很容易从一个线程转移到另外一个线程。

当转移到分布式系统时，情况发生了变化。在一个分布式系统中，仍然有并发性，因为不同计算机可以并行地执行程序。然而，通常没有共享内存，因为分布式系统中的每台计算机都运行自己的操作系统，有自己的地址空间，使用自己的内存。不同的计算机只能使用网络相互发送信息来进行通行。（在一些超级计算机和研究系统中存在有限形式的分布式共享内存，还有像RDMA(remote direct memory access)技术，通过网络访问对方的内存。此外，数据库在某种意义上也可以被视为共享内存，与字节序寻址相比，它的数据模型不同。广义上讲，大多数实用的分布式系统都是基于消息传递的。



## 系统模型

系统模型用于描述我们对节点和网络行为方式的假设。它是对其属性的抽象描述，在实践中可以通过不同技术来实现。

为了说明分布式系统模型，我们将从分布式系统中两个经典的思维实验开始：两将军问题和拜占庭将军问题。

### 两将军问题

在两将军问题中，两位将军分别带领一支军队，他们相占领一座城市。这座城市的防御性很强，如果两只军队中只有一支军队进攻，那么军队会被击退。除非两只军队同时进攻，才能成功占领。

因此，两位将军需要协调他们的进攻计划。由于两支军队距离很远，而且他们只能通过信使沟通，这使得协调工作变得很困难。信使必须通过城市控制的领土，他们有时会被敌军抓住。因此，一个将军发出的消息可能被另外一个将军收到，也可能收不到，发信息的人不知道他的消息是否得到传递，除非对方给出明确回复。如果一个将军没有收到任何消息，他不知道这是因为信使被俘虏了还是因为另一个将军没有发送任何消息。

<div class="row">
    <div class="col-sm mt-3 mt-md-0">
        {% include figure.html path="assets/img/image-20230508160700122.png" title="grad" class="img-fluid rounded z-depth-1" %}
    </div>
</div>


那么两位将军改用什么协议来制定一个计划呢？对两位将军来说，有两种选择方案：

+ 要么将军承诺在任何情况下都会进行攻击（即使没有收到回应）
+ 要么将军在承诺攻击之前等待确认。

将军们如何做决定？

1. 将军1总是攻击，即使没有收到回应？
   + 发送大量的信使，增大消息传出去的概率
   + 如果信使全部被俘，将军2不知道将军1的攻击消息，那么将军1就会被击退
2. 将军1等待将军2的回应才攻击？
   + 将军1是安全的
   + 但将军2知道，只有将军1收到他的回信时才能进攻
   + 现在，将军2的情况与第一种情况中的将军1是相同

**没有共同的知识：获取某事的唯一方法是沟通**。问题是无论交换多少消息，两位消息都无法确定另外一只军队也会在同一时间出现。一些列重复来回确认可以建立其逐渐增加的信心（即将军们的意见是一致的），但可以明确的是他们无法通过交换任何有限数量的信息来传达确定性。

这个思维实验表面，在一个分布式系统中，一个节点没有办法确定另外一个节点的状态。一个节点如何知道一些事情的唯一办法是通过消息传达。

| 商家 | 银行 | 结果 |
| :-: | :-: | :-: |
| 不发货 | 没收钱 | 没有交易发生 |
| 发货 | 没收钱 | 商家血亏 |

| 发货 | 收钱 | 完整的交易 |

将网上购物改编为两将军问题的一个实际例子。商店和信用支付服务通过RPC进行通信，其中一些消息可能会丢失。然后，商家希望确保只有在商品被支付后才发货，并且只有在货物被发送后才向用户卡收费。

实际上，这并完全符合两将军问题模型：先给钱在收货其实是安全的，因为商店最终不能发货，它可以退钱，也可以取消付款来解决问题。如果商店和支付服务之间的通信中断，商店可以等待，知道连接恢复，然后查询并确定交易的状态。



### 拜占庭将军问题

    <div class="col-sm mt-3 mt-md-0">

    </div>

类似于两将军问题的背景。我们再次假设，有三个或更多的军队想要占领一座城市，将军们通过信使进行交流，这次我们假设发出的消息一定可以被送达。

拜占庭问题的难点在于一些将军可能是 “traitors”（叛徒），也就是说，他们可能将恶意地误导其他将军。我们称这些叛徒为malicious，而其他人则是 honest.

<div class="row">
    <div class="col-sm mt-3 mt-md-0">
        {% include figure.html path="assets/img/image-20230508164048740.png" title="grad" class="img-fluid rounded z-depth-1" %}
    </div>
</div>


举个例子，将军3从将军2和将军1收到了两条相互矛盾的消息。将军1告诉将军3将要进攻，而将军2则声称将军1要撤退。将军3不可能确定：

+ 将军2是否在撒谎（情况1）
+ 或者将军2是诚实的，但将军1却发出了矛盾的命令（情况2）

<div class="row">
    <div class="col-sm mt-3 mt-md-0">
        {% include figure.html path="assets/img/image-20230508164745368.png" title="grad" class="img-fluid rounded z-depth-1" %}
    </div>
</div>


拜占庭将军问题：

+ 每个将军要么是恶意的，要么是诚实的
+ 最多可以有 f 名恶意的将军
+ 诚实的将军不知道谁是恶意的人
+ 恶意的将军可能会勾结在一起
+ 然而，诚实的将军们必须就计划达成一致



我们推导出一些结论：

+ **定理**：总共 $3f + 1$ 位将军才能容忍 $f$ 名恶意的将军
+ 密码学（数字签名）有帮助，但问题仍然难以解决

诚实的将军不知道谁是恶意的将军，但是恶意的将军可以勾结起来，秘密地协调他们的行动，我们甚至可以假设，所有的恶意将军都被一个邪恶对手所控制。那么拜占庭将军问题就是要确保所有诚实的将军都同意相同的计划（进攻还是撤退）。根据定义，我们不可能知道恶意的将军要做什么，所以我们能做的最好的事情就是让诚实的将军们达成一致。

事实上，这是很困难的：在一个具有恶意将军和不可预测的通信延迟系统中，可以证明拜占庭将军问题只有严格在少于三分之一的将军是恶意的情况下才能得到解决。

拜占庭将军问题是否具有实际意义？

<div class="row">
    <div class="col-sm mt-3 mt-md-0">
        {% include figure.html path="assets/img/image-20230508170632914.png" title="grad" class="img-fluid rounded z-depth-1" %}
    </div>
</div>


真正的分布式系统确实涉及复杂的信任关系。例如，客户需要信任一家网上商店能够真正地交付他们订购的商品，尽管他们可以通过银行对付款提出异议。但是，如果一家网点以某种法师允许顾客在不付款的情况下订购商品，这个弱点无疑会被欺诈者所利用，所以网点必须假设顾客是潜在的恶意的。

另一方面，对于属于商店的服务之间的RPC，运行在同一个数据中心，一个服务可能可以信任由同一公司运行的其他服务。支付服务并不完全信任商店，因为有人可能建立一个欺诈性的商店或使用被盗的信用卡，但商店可以确实信任支付服务。

拜占庭将军问题是对于这种复杂信任关系的简化，但它是研究一些参与者可能有恶意行为的系统的一个好起点。

在分布式系统中，一些系统明确地处理了一些节点可能被恶意控制的可能性，这样的系统被称为**拜占庭容错**。



### 节点和网络行为

    <div class="col-sm mt-3 mt-md-0">

    </div>

当设计一个分布式算法时，system model 系统模型是我们对可能发生故障的假设性具体描述。

我们已经看到了两个思维实验：

+ 两将军问题：网络通信模型
+ 拜占庭将军问题：节点行为模型

在实际的系统当中，节点和网络都有可能有问题。

系统模型中呈现的假设由以下内容组成：

+ 网络行为（如信息丢失）
+ 节点行为（如崩溃）
+ 计时行为（如延迟）

每一部分都有对应不同的模型选择。

#### 网络

没有一个网络是完全可靠的，即使在一个精心设计的具有冗余网络链接的系统中，也可能出错。有人可能意外地拔掉网线，鲨鱼和奶牛对长距离的网络造成损害和中断，或者一个网络可能暂时过载，也许是意外，也许是Dos攻击。任何这些情况都有可能导致信息丢失。

在系统模型中，我们采取更抽像的措施，不必在意鲨鱼和奶牛这些细节。大多数分布式算法假设网络在一对节点之间提供双向的信息传递，point-to-point 或者 unicast 单播通信。真是的网络确实允许广播或者多播通行。但基本上，假设现在的互联网只有单播比较好。（在后续章节，或展示如何在单播基础上实现广播）

    <div class="col-sm mt-3 mt-md-0">

我们要分辨这些假设链接的可靠程度。大多数算法基于以上所列的三种之一（假设两节点之间是双向的 point-to-point 通信）：

+ Reliable 可靠链接
  + 发送出的消息一定被接收
  + 信息可能会被重新排序
+ Fair-loss 公平损耗链接
  + 信息可能会丢失，重复，或重新排序
  + 重试 + 扣除
  + 不断重试可以确保信息最终抵达
+ Arbitary 任意链接
  + 存在一个恶意的对手，他可能会干扰信息（窃听、修改、丢弃、欺骗、重放）

网络分区：部分链接上，所有消息掉线或者延迟一定时间

我们可以根据一些方法将某一类型的链接转换成其他类型。例如我们有一个**公平损耗**链接，可以通过不断重传丢失的信息，知道最终收到这些信息，将其变成一个**可靠**链接。公平损耗假设意味着网络分区只会存在有限的时间，而不是永远，所以我们可以保证每条消息最终会被收到。在网络中断期间发送的任何消息只有在中断修复后才能收到，这可能需要很长时间。

TCP协议在网络数据包层面实现了这种重传和去重。然而，TCP通常会配有一个 timeout 超时，因此会在一定时间后放弃并停止重试。为了克服更长时间的网络中断，除了TCP提供的机制外，还需要实施额外的重传和去重机制。

**[练习]** 假设你有一个客户端-服务端的RPC系统，其中客户端会重复一个RPC请求，知道收到响应。服务器怎样才能对客户端请求去重？

**[一个解答]** 服务器可以为每个RPC请求分配一个唯一标识符，当客户端发送一个RPC请求时，将该标识符包含在请求中。服务器可以将每个标识符存储在一个已处理的请求列表中，以便在收到重复请求时对齐进行比较。如果请求在表中，则说明该请求已经被处理，可以忽略，如果不在，则响应并将其加入已请求表中。在一定时间后，可以将标识符从列表中移除，以便允许相同的请求在未来再次处理。

**任意链接**才是准确描述互联网上通信的模型：只要你的通信通过一个网络（无论是星巴克的WIFI还是互联网主干网络），该网络的运营商都有可能以任意方式干扰和操作你的网络数据包。操作网络流量的人被认为是**主动攻击者** active adversary。幸运地是，使用加密技术将**任意链接**变为**公平损耗链接**几乎是可能的。运输层安全协议（TLS），防止主动攻击者窃听、修改、欺骗或重放流量。TLS唯一不能防止的是攻击者阻断通信。因此，只有在我们假设对手永远不会阻断通信的情况下，才能将任意链接转化成公平损耗链接。

因此，可靠链接并非完全不现实。一般来说，只要在网络中断期间等待一段重试时间，所有发送的消息都有可能被收到。然而，我们还必须考虑这样一种可能性，当消息发送者在试图重传消息时可能宕机，这可能导致该消息永久丢失。因此，这需要进行到节点崩溃这一话题讨论。

#### 节点行为

<div class="row">
    <div class="col-sm mt-3 mt-md-0">
        {% include figure.html path="assets/img/image-20230508200732930.png" title="grad" class="img-fluid rounded z-depth-1" %}
    </div>
</div>


每个节点执行一套特定的程序，假设符合以下之一：

+ crash-stop 崩溃-停止（故障-停止）
  + 在任何时候，如果一个节点崩溃，那么一定发生异常。崩溃后，该节点彻底停止运行
+ crash-recovery 崩溃-恢复（故障-恢复）
  + 一个节点可能在任何时候崩溃，并丢失内存状态。它可能在以后的某个时候恢复执行。存储在磁盘上的数据崩溃后依然存在。
+ 拜占庭式（故障-任意行为）
  + 如果一个节点偏离了原有程序，它一定有异常发生。异常的节点可能做任何事情，包括崩溃或者恶意行为。

没有任何异常的节点被称为 **”correct“**

在崩溃-停止的模型中，我们假设一个节点崩溃后，永远不会恢复。对于无法恢复的硬件故障来说，这是合理的，或者对于一个手机掉进厕所的人来说，这也是一个合理的模型。对于软件崩溃，崩溃-停止模型可能显得不那么现实，因为我们可以直接重启节点来恢复。尽管如此，一些算法还是假设程序符合崩溃-停止模型，因为这使得算法更简单。在这种情况下，一个崩溃后恢复的节点将被看作一个重新加入系统的新节点。

崩溃-恢复模型明确地表示允许节点在崩溃后重启并恢复。当一个节点崩溃重启时，我们假设它的所有内存状态都会丢失，但它在磁盘上存储的数据都会被保留下来。该模型没有假设节点崩溃后可能需要多长时间才能恢复，而且崩溃的节点有可能永远不会恢复。

拜占庭模型是最通用的节点行为模型。如拜占庭将军问题，有故障的节点不仅可能崩溃，而且可能以任意形式的方式偏离指定的程序，包括表现出恶意行为。一个节点的实现中的错误也可以被归类为拜占庭式故障。然而，如果所有节点都运行相同的软件，那么他们都会有相同的错误。因此，容错的前提是不超过三分之一的节点有拜占庭式故障。原则上，我们可以尝试使用同一种算法的几个不同实现，但这个选择很不现实。因此，我们通常在描述程序意图偏离协议时使用拜占庭这一术语，而不是 bug。

在网络的情况下，可以使用通用协议将一个模型转化成另外一个模型。而在节点行为的不同模型中，情况就不是这个样了。如，崩溃-恢复系统模型设计的算法可能看起来与拜占庭算法完全不相关。

#### 时间

    <div class="col-sm mt-3 mt-md-0">

这一部分是同步性假设，也就是关于时间。我们在这里有三种选择：**同步**，**异步**，或者**部分同步**

假设网络和节点有以下情况之一：

+ 同步
  + 消息延迟不超过已知上限
  + 节点以已知速度执行算法
+ 部分同步
  + 系统在某些有限的（但未知）时间段异步，其余时间内是同步的
+ 异步
  + 消息可以被任意延迟
  + 节点可以被任意地执行暂停
  + 完全没有时间（计时）保证

注意：计算机科学其他领域以不同的方式使用 “同步” 和 “异步” 术语。如在RPC和IO操作语境中，“同步”意味着“调用者阻塞/等待操作完成”， 而“异步”意味着“调用者在发出请求后不用等待，继续执行”。

同步系统是大家最喜欢的，通过网络发送消息永远不会超过某个已知的最大延迟，而且节点总是以可预测的速度执行算法。如果你假设一个同步系统，分布式计算中许多问题就容易得多。而且，同步假设很上头，因为网络和节点在大多数时候都不会出错的，所以这个假设在 most of time 大多数时候为真。

但是 most of time 大多数时候并不是 always 总是这样。一旦违反有限延迟和可预测执行速度的假设（即使是很短暂）。基于同步模型设计的程序往往会出现灾难性的崩溃。在实际的系统中，有太多原因导致网络延迟或执行速度出现变化。

另外一个极端模型是异步模型，在这个模型中，我们根本不做时间假设，我么允许消息在网络中任意延迟，我么允许节点处理速度的任意差异（如一个节点暂停执行而其他节点正常运行）。为异步模型设计的算法通常非常稳健，因为它们不受任何临时网络中段或者延迟峰值影响。不幸地是，分布式计算中一些问题在异步模型中无法解决

因此，我们有部分同步模型作为折衷的方案。在部分同步模型下，我们假设我们的系统在大多数时候是运行良好的的，但偶尔也会转为异步模式。异步模式下没有任何时间保证，而且无法预测这种情况什么时候发生。部分模型对许多实际的系统很友好，但需要正确且谨慎地使用。

违反同步假设的原因有很多。网络通常有一定不可预测的延迟，以下场景都会增加延迟：

+ 信息丢失，导致需要重试
  + 信息丢失并重传，延迟会持续增加，特别是如果我们必须等待网络中断修复后，继续传信息。
+ 拥堵/拥挤导致排队
  + 网络中延迟增加的另外一个原因是拥堵，导致数据包在交换机缓冲区排队。
+ 网络/路线重新配置
  + 网络重新配置也会导致大量延迟

我们可能期望程序在某个节点上执行的速度是恒定的，毕竟一条指令一般需要固定数量的CPU时钟周期，而时钟速度本身不会有太大变化。然而，即使在单一节点上，也有许多原因导致正运行的程序被以外暂停了很长时间。

+ 操作系统调度问题，如优先级变动
  + 系统中的调度可以抢占一个正在运行的线程，让它在其他程序运行时暂停，特别是在负载很重的机器上
+ 基于 Stop-the-world （STW全局暂停）的垃圾回收（JVM Full GC)
  + 这是一个实际的问题，在Java这样的内存管理语言中，执行垃圾回收是，它需要不时暂停执行的线程。
+ 页面故障（Page fault）、交换、颠簸
  + 页面故障是一个线程可能被暂停的另外一个原因，特别是当剩余可用内存不多的时候。

实时操作系统（RTOS）提供调度保证，但大多数分布式系统不使用 RTOS.

在执行程序的任意阶段，线程可能且一定会在最不方便的时刻被抢占。在分布式系统中，这尤其成为问题，因为对于一个节点来说，当它被暂停时，时间似乎 “stand still 静止”。而在这段时间里，所有节点继续正常执行他们的算法。其他节点甚至可能注意到暂停的节点没有响应并认为它宕机了。一段时间后，暂停的节点恢复处理，甚至没意识到它已经暂停很长时间了。

这些执行暂停与之前讨论的崩溃和重启不一样。当一个正在执行的进程或线程被暂停时，它通常不会注意到它被暂停了，除非它定期检查系统时钟来测量已过去的时间。另一方面，重启是由程序明确处理的，因为它的内存状态在崩溃时丢失了，在重启时它可能会从磁盘加载其持久状态。

再加上网络延迟变化的原因，这意味在实际的系统当中，假设一个同步模型是非常不安全的。大多数分布式算法需要为异步或者部分同步模型设计。

### 容错和高可用性

从商业角度看，通常最重要的是可用性。比如网站，一家网上商店希望能够在白天或者晚上的任何时候销售产品，网站的任何中断都意味着失去了赚钱的机会。

服务的可用型通常是以其在一定时间内正确响应请求的能力来衡量的。一个服务是“可用”还是“不可用”的定义可能有些武断：例如，如果加载一个页面需要5s，我们是否仍然认为该网站可用？如果它需要30s呢？一个小时呢？

通常，服务的可用性期望的正式定义为 **service-level objective 服务水平目标**（SLO），它通常规定了由某个客户在某段时间内某，需要（在指定的timeout内）正确响应的百分比。**service-level- agreement 服务水平协议**是一个合同，它规定了一些SLO，以及如果没有达到SLO的后果（如服务提供商向用户退款）。

故障（如节点崩溃或者网络中断）是导致不可用的一个常见原因。为了提高可用性，我们可以减少故障的频率，或者我们可以设计系统使其在某些部件出现故障的情况下继续工作。后一种方法被称为 **fault tolerance 容错**。通过购买更高质量的硬件和引入冗余来减少故障频率是可行的，但这种方法无法将故障的概率降低到零。所以许多分布式系统采取容错的方法。

容错性总是相对于可以容忍的最大故障数量。例如，一些分布式算法在少于一般的节点崩溃情况下依然能运行，但如果超过一半节点崩溃就会停止工作。想要容忍无限数量的故障是不现实的：如果所有节点都崩溃并无法恢复，那么无论多么高明的程序都无法运行。

在某些系统中，一个组件出现故障会导致整个系统的宕机。这样的组件被称为 single point fault单点故障（SPOF），而容错系统一般都试图避免出现任何单点故障。例如，互联网被设计成没有单点故障：没有一个服务器或者路由器崩溃而导致整个互联网瘫痪。

#### 故障检测

<div class="row">
    <div class="col-sm mt-3 mt-md-0">
        {% include figure.html path="assets/img/image-20230509151143894.png" title="grad" class="img-fluid rounded z-depth-1" %}
    </div>
</div>


+ 故障检测器：检测另外一个节点是否有故障的算法
+ 完美故障检测器：当且仅当一个节点崩溃是，该节点被标记为故障。
+ 崩溃-停止/崩溃-恢复的典型实现：发送消息，等待响应，如果在某个超时的时间内没有响应，则将节点标记为故障。
+ 问题：无法区分崩溃的节点、暂时无响应的节点、丢失的信息和延迟的消息。

容错的第一步是故障检测，这通常是通过 **failure detector 故障检测器**完成的。故障检测器通常检测的是崩溃故障。拜占廷故障不总是可以检测到的，尽管在某些情况下拜占廷行为确实留下了线索，可以用来识别和排除恶意节点。

在大多数情况下，故障检测器的工作方式是定期向其他节点发送消息。如果在预期的时间内没有收到响应，就将该节点标记为崩溃。理想情况下，我们希望只有当节点真的崩溃时才发生超时（这被成为完美故障检测器）。然而，两将军问题告诉我们，这并不是一个完全准确的检测崩溃的方法，因为没有响应也可能是由于消息丢失或延迟。

<div class="row">
    <div class="col-sm mt-3 mt-md-0">
        {% include figure.html path="assets/img/image-20230509152156864.png" title="grad" class="img-fluid rounded z-depth-1" %}
    </div>
</div>


完美的基于超时的故障检测只存在于具有可靠链路、同步的、崩溃-停止系统中。在部分同步系统中，一个完美故障检测器并不存在。此外，在异步系统中，不存在基于超时的故障，因为在异步模型中超时是没有意义的。然而，在部分同步系统中存在一个有用的故障检测器：**eventually perfect failture detector 最终完美故障检测器[1996]**。

最终完美故障检测器：

+ 可能暂时将一个节点标记为崩溃，尽管他是正确的
+ 可能暂时将一个节点标记为正确，即使他已经崩溃了
+ 最终，当且仅当一个节点已经崩溃时，将其标记为崩溃

以上反映了故障检测并不是即时的，可能存在错误的超时。在后面j介绍中可以看到，如何使用这样的故障检测器来设计容错机制，并自动从节点崩溃中恢复。使用这样的算法，就有可能建立高度可用的系统。容错设计也使日常操作更容易：例如，如果一个服务可以容忍三个节点中的一个不可用，那么就可以通过安装软件升级、并一次重启一个节点的方式来升级软件，同时其余来那个个节点继续运行该服务。以这种方式推出软件升级，客户端不会注意到任何中断，这类发布方式对许多无间断升级软件的组织来说是非常重要的。

对于 safety-critial 安全关键应用，如空中交通控制系统，良好的容错机制无疑是非常重要的。然而，并不是可用性越高就与越好。要达到极高的可用性，需要集中很多的工程资源，而且往往选择保守的设计。在某些时候，实现高可用性的额外成本超过了偶尔停机的成本，所以接受一定量的停机时间在经济上是合理的。
